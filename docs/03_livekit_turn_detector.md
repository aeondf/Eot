# Вариант 2: LiveKit turn-detector multilingual

## Что это

Текстовый классификатор от LiveKit. Берёт транскрипт от ASR и говорит — реплика завершена или нет.

Модель: Qwen2.5-0.5B-Instruct, файнтюнена на conversational данных, экспортирована в ONNX INT8.

## Как работает

```
текст реплики + контекст (последние 4 хода диалога) → Qwen2.5 encoder → score (0..1)
```

Понимает смысл: "мне нужно во-первых..." — явно incomplete, даже если пауза длинная.

## Плюсы и минусы

- [+] Понимает смысловую завершённость
- [+] 14 языков включая русский
- [+] ONNX INT8 — быстрый CPU-инференс
- [+] Учитывает контекст диалога (последние ходы)
- [-] Нужен транскрипт от ASR — дополнительная задержка (ждём Qwen)
- [-] Качество зависит от качества транскрипции (ошибки ASR -> ошибки классификатора)
- [-] Не ловит интонацию и тд (только текст)

## Что на входе

- Строка текста — накопленный транскрипт текущей реплики
- Опционально: история диалога (agent + user turns)
- Язык определяет ASR провайдер, передаём в модель

## Где взять

- HuggingFace: [livekit/turn-detector](https://huggingface.co/livekit/turn-detector)
- PyPI: `pip install livekit-plugins-turn-detector` (deprecated) → сейчас в `livekit-agents`
- Модель: `model_q8.onnx` — можно тащить отдельно без livekit SDK

> Важно: мультиязычная модель лежит **не в `main`**, а в ревизии `v0.4.1-intl`
> (`onnx/model_q8.onnx` + `languages.json`). В main — старая English SmolLM2,
> работает иначе. ONNX уже отдаёт готовую вероятность `prob[batch, seq]` —
> берём последнюю позицию, softmax не нужен.

## Результаты прогона (спайк)

Прогон на 3 датасетах, метки через обрезку. Ключевая находка —
**домен-чувствительность**:

| Датасет | Домен | ROC-AUC |
|---|---|---|
| `d0rj/dialogsum-ru` | разговорные бытовые диалоги (≈колл-центр) | **0.879** |
| `inkoziev/Conversations` | книжные диалоги | 0.81 |
| smart-turn (транскрипт Whisper) | начитка / энциклопедия | 0.43 |

- Модель работает только на **разговорном** тексте; на начитке проваливается
  (хуже случайного) — не считает длинную фразу завершённым ходом диалога.
- **Масштаб score низкий**: complete в среднем ~0.2–0.3, incomplete ~0.06.
  `score > 0.9` не сработает — рабочий диапазон в районе сотых-десятых.
- Рекомендуемый порог **~0.2** (на dialogsum-ru: precision 0.84, recall 0.64).
- Latency ~10 мс на CPU (растёт с длиной текста).

Стартовый порог переподобрать на реальных размеченных звонках. Дефолт LiveKit
`0.0032` не брать — он под их полный пайплайн с таймингами тишины.
