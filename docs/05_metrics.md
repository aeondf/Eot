# Метрики для оценки подходов

## Общие метрики (для всех 3 вариантов)

| Метрика | Что это | Важность |
|---|---|---|
| **FP (False Positive)** | Отправили в LLM, а человек не закончил | 🔴 Критично — пользователь получает ответ посередине фразы |
| **FN (False Negative)** | Не отправили, хотя закончил | 🟡 Высокая — лишняя задержка UX |
| **Latency** | Сколько мс от конца последнего слова до отправки в LLM | 🔴 Критично для колл-центра |
| **F1 / ROC-AUC** | Общее качество классификатора | 🟡 Для сравнения моделей между собой |

**Главный trade-off**: снижаем FP — растёт FN (ждём дольше). Для колл-центра FP важнее — лучше чуть подождать, чем оборвать человека.

---

## Вариант 1: Pipecat Smart Turn v3 (аудио)

**Что меряем:**
- Score (0..1) на русских аудио с завершёнными и незавершёнными репликами
- При каком пороге F1 максимальный
- FP/FN при нашем пороге 0.9
- Latency инференса на нашем GPU

**Специфичная проблема:** модель не знает смысла — только звук. Надо проверить:
- Пауза внутри перечисления ("мне нужно... номер карты...")  → должно быть incomplete
- Вопрос с восходящей интонацией → должно быть complete
- "Угу", "ага" посередине → должно быть incomplete

**Источник тестовых данных:** записи звонков (или синтетика через TTS).

---

## Вариант 2: LiveKit turn-detector (текст)

**Что меряем:**
- Score на тестовых транскриптах (complete / incomplete примеры)
- Насколько устойчив к ошибкам ASR (специально добавляем ошибки транскрипции)
- Latency: время от получения транскрипта до score

**Специфичная проблема:** зависит от Qwen ASR. Надо проверить:
- Qwen плохо расставляет пунктуацию → влияет ли это на score
- Неполный транскрипт в реальном времени → как модель ведёт себя на частичных словах
- Контекст диалога (последние 4 хода) — даёт ли это реальный прирост качества

**Источник тестовых данных:** транскрипты из Qwen на тех же аудио что и вариант 1 (сравниваем яблоки с яблоками).

## Вариант 3: Файнтюн (если до него дойдём)

**Что меряем после обучения:**
- F1 на held-out тесте (20% датасета)
- Сравнение с вариантами 1 и 2 по FP/FN
- Latency инференса (должен быть не хуже чем у исходной модели)

**Специфично:**
- Переобучение: смотрим на разрыв train/val loss
- Качество на out-of-domain данных (не только колл-центр)

---

## Как будем тестить

1. Набираем ~50 тестовых примеров вручную: записи или синтетика, 25 complete / 25 incomplete
2. Прогоняем через каждую модель, записываем score
3. Строим confusion matrix при пороге 0.9
4. Смотрим на граничные случаи (score 0.5-0.8) — что там за реплики

**Победитель** — тот у кого минимум FP при FN < 20% и latency < 300ms.
